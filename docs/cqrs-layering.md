# CQRS / DDD 分层与 handler 职责

## 各层职责

| 层                         | 做什么                                                    | 不做什么                     |
| -------------------------- | --------------------------------------------------------- | ---------------------------- |
| Controller / transport     | 反序列化 → command/query → 派发;把领域异常映射成 HTTP/SSE | 不写逻辑                     |
| 输入校验(pipe / schema)    | command **形状**检查(必填、类型、格式)                    | 不碰领域状态                 |
| **Command handler = 用例** | 加载聚合 → 调聚合方法 → 持久化 → 发事件;返回 ack/id       | 不写业务规则、不跨聚合改状态 |
| **聚合 / 实体**            | 不变量、业务规则、状态迁移;抛领域异常                     | 不碰 I/O                     |
| 领域服务                   | 跨聚合、不属于任一聚合的逻辑                              | 不持久化                     |
| Repository                 | 持久化 + 身份/归属过滤(租户、owner)                       | 不写业务规则                 |
| Query handler / 读模型     | 纯查询、为读优化、无规则                                  | ——                           |
| Saga / 进程管理            | 跨聚合、跨 BC、长流程,经事件驱动                          | ——                           |

handler 该厚该薄不是重点——Vernon / Axon / NestJS-CQRS 体系里,handler **本就**承担用例编排(它常常就是 application service 本身)。重点是它**不该做什么**。

## 例子：两种「校验」——这是最常踩的坑

|               | A. 输入 / 形状校验                                                  | B. 领域 / 不变量校验                                |
| ------------- | ------------------------------------------------------------------- | --------------------------------------------------- |
| 问的是        | 「这个命令本身成立吗?」                                             | 「就当前系统状态,这操作允许吗、有意义吗?」          |
| 依赖领域状态? | 否                                                                  | **是**                                              |
| 例子          | 必填、类型、id 格式、非空                                           | 已发货不能取消、用户不拥有此会话、未激活不能开 turn |
| 归属          | **边界**:DTO schema / class-validator / ajv,理想在 handler **之前** | **聚合或领域服务**                                  |
| 能进 handler? | ✅ 可以(且最好在 handler 前)                                        | ❌ 不行                                             |

「想把校验搬进 handler」之前,先分清是 A 还是 B。A 搬进/靠近 handler 是对的;B 搬进 handler 就是数据袋陷阱。

### 授权不变量住哪儿

「这用户拥不拥有这个资源」属于 B,有两个合法的家(二选一,看口味):

```ts
// 方案 1:聚合方法
conversation.requireOwnedBy(userId); // 不满足就抛 ConversationForbiddenError

// 方案 2:repository 归属过滤(顺带防信息泄露:不暴露「存在但不是你的」)
const conv = await repo.findById(id, userId); // 不属于 → null
if (!conv) throw new NotFoundError(id);
```

**真正的坏味道不是「检查写在 handler 里」,而是「同一不变量散落、各 handler 各凭良心」**——必然漏。集中到聚合或 repo,所有路径 transitively 受保护,这才是 DDD。handler 合法持有的只有 A 类输入校验,以及把领域异常**翻译**成 401/403/404 的那一步映射——**抛**在领域,**映射**在边界。

## 例子：用例无归属(Anti-pattern)

当一个用例跨多个聚合 / BC,而**没有任何 service 拥有它**时,它的逻辑就会被 smeared(涂抹)到一串 handler 里。典型症状:某条业务规则(「归档邮件要建一个会话」「turn 结束后压不压历史」)只出现在某个 handler 的私有方法里,别处复现不了。

正解:**给用例命名一个家**——要么是一个 application service 方法(`ChatService.startTurn()`),要么,当它跨聚合 / 跨 BC / 长流程时,是一个**显式的 saga / 进程管理器**。让 handler 退回成薄步骤。

## handler 决策清单

往 handler 里放某段逻辑前,问一句:

> **「这条规则换个入口——定时任务、别的用例、事件回调——还成不成立?」**
>
> 还成立 → 它是业务规则,不该在 handler。
> 不成立(只对「这个 HTTP 请求」有意义)→ 它是传输 / 编排,可以留。

再问一句:

> **「它依赖当前领域状态吗?」**
>
> 依赖 → 是 B 类不变量,进聚合 / 领域服务。
> 不依赖 → 是 A 类输入校验,进 DTO schema / 边界。

## 命令 vs 事件

- **Command handler**:执行一个意图,改状态,返回 ack。命令**会被拒绝**(抛异常)。
- **Event handler**:对一个**已发生**的事实做反应(投影到读模型、触发下游 saga 步骤、发 SSE 帧)。事件不拒绝——它是历史,只管 consequences。

不要把「应该是一个用例」的逻辑塞进 event handler 去做「顺便」;也不要让 command handler 去订阅事件。两者语义不同,长流程归 saga。
